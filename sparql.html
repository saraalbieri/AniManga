Validazione tramite SPARQL Queries

Le query SPARQL sono state usate prima per esplorare il dominio anime/manga in Wikidata e solo successivamente per verificare la proprietà wdt:P9071. In questo modo il gap sugli archetipi narrativi non è stato assunto in partenza, ma individuato progressivamente confrontando le proprietà disponibili sui personaggi.

Fase esplorativa 1: trovare le opere su Wikidata

La prima operazione è stata cercare le entità Wikidata relative alle tre opere scelte: One Piece, Naruto e Death Note. Questa query serve a verificare che il dominio sia già rappresentato nel Knowledge Graph.

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX schema: <http://schema.org/>
PREFIX wikibase: <http://wikiba.se/ontology#>
PREFIX bd: <http://www.bigdata.com/rdf#>

SELECT DISTINCT ?work ?workLabel ?description
WHERE {
  VALUES ?work { wd:Q710324 wd:Q25929253 wd:Q718624 }
  OPTIONAL { ?work schema:description ?description . FILTER(LANG(?description) = "en") }
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en,it". }
}
ORDER BY ?workLabel

Perché è utile: mostra che le opere sono già presenti in Wikidata e possono essere usate come punto di partenza per esplorare i personaggi collegati.

Fase esplorativa 2: recuperare i personaggi collegati alle opere

Dopo aver individuato le opere, abbiamo cercato i personaggi collegati. La query usa UNION perché Wikidata può rappresentare il legame personaggio-opera con proprietà diverse, come P1441 e P1080.

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT DISTINCT ?character ?characterLabel ?work ?workLabel
WHERE {
  VALUES ?work { wd:Q710324 wd:Q25929253 wd:Q718624 }
  { ?character wdt:P1441 ?work . }
  UNION
  { ?character wdt:P1080 ?work . }
  ?work rdfs:label ?workLabel .
  ?character rdfs:label ?characterLabel .
  FILTER(LANG(?workLabel) = "en")
  FILTER(LANG(?characterLabel) = "en")
}
ORDER BY ?workLabel ?characterLabel
LIMIT 100

Perché è utile: questa query ci ha permesso di vedere che Wikidata contiene già numerosi personaggi delle tre opere, quindi il problema non è l'assenza delle entità.

Fase esplorativa 3: osservare le proprietà dei personaggi principali

Prima di concentrarci su P9071, abbiamo estratto le proprietà disponibili per alcuni personaggi principali: Monkey D. Luffy, Naruto Uzumaki e Light Yagami. L'obiettivo era capire che tipo di informazione Wikidata registra più frequentemente.

SELECT DISTINCT ?character ?characterLabel ?property ?propertyLabel ?value ?valueLabel
WHERE {
  VALUES ?character { wd:Q477948 wd:Q931 wd:Q843545 }
  ?character ?property ?value .
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en,it". }
}
ORDER BY ?characterLabel ?propertyLabel
LIMIT 300

Risultato metodologico: questa esplorazione ha mostrato che le proprietà disponibili sono soprattutto enciclopediche, identificative o editoriali. Da qui è nata la domanda: Wikidata rappresenta anche la funzione narrativa e l'archetipo dei personaggi?

Dal confronto delle proprietà al gap

Dopo aver confrontato personaggi principali e secondari, abbiamo cercato una proprietà adatta a rappresentare gli archetipi narrativi. La proprietà individuata è wdt:P9071, character type. Il passaggio successivo è stato quindi verificare se questa proprietà fosse effettivamente valorizzata nei personaggi analizzati.

Conclusione della fase esplorativa: il vocabolario esiste, ma il dato manca o è incompleto. Questo è il gap semantico su cui si basa il progetto.

Query 1: personaggi selezionati e valori P9071

Questa query usa i QID verificati dei 15 personaggi e controlla, tramite OPTIONAL, se esiste un valore per wdt:P9071.

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
PREFIX wikibase: <http://wikiba.se/ontology#>
PREFIX bd: <http://www.bigdata.com/rdf#>

SELECT ?character ?characterLabel ?characterType ?characterTypeLabel
WHERE {
  VALUES ?character { wd:Q477948 wd:Q858432 wd:Q877964 wd:Q1061765 wd:Q1190653 wd:Q931 wd:Q1740 wd:Q203240 wd:Q1767 wd:Q1043344 wd:Q843545 wd:Q52863 wd:Q687038 wd:Q2328583 wd:Q138008 }
  OPTIONAL { ?character wdt:P9071 ?characterType . }
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en,it". }
}
ORDER BY ?characterLabel ?characterTypeLabel

Query 2: personaggi collegati ai tre anime

Questa query usa UNION perché Wikidata può collegare i personaggi alle opere attraverso proprietà diverse. Le tre opere sono selezionate tramite QID verificati, così evitiamo ambiguità dovute alle label.

PREFIX wd: <http://www.wikidata.org/entity/>
PREFIX wdt: <http://www.wikidata.org/prop/direct/>
PREFIX rdfs: <http://www.w3.org/2000/01/rdf-schema#>

SELECT DISTINCT ?character ?characterLabel ?work ?workLabel
WHERE {
  VALUES ?work { wd:Q710324 wd:Q25929253 wd:Q718624 }
  { ?character wdt:P1441 ?work . }
  UNION
  { ?character wdt:P1080 ?work . }
  ?work rdfs:label ?workLabel .
  ?character rdfs:label ?characterLabel .
  FILTER(LANG(?workLabel) = "en")
  FILTER(LANG(?characterLabel) = "en")
}
ORDER BY ?workLabel ?characterLabel
LIMIT 100

Query 3: personaggi senza P9071

Questa query usa FILTER NOT EXISTS per isolare i personaggi del campione privi di archetipo narrativo strutturato.

SELECT ?character ?characterLabel
WHERE {
  VALUES ?character { wd:Q477948 wd:Q858432 wd:Q877964 wd:Q1061765 wd:Q1190653 wd:Q931 wd:Q1740 wd:Q203240 wd:Q1767 wd:Q1043344 wd:Q843545 wd:Q52863 wd:Q687038 wd:Q2328583 wd:Q138008 }
  FILTER NOT EXISTS { ?character wdt:P9071 ?characterType . }
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en,it". }
}
ORDER BY ?characterLabel

Interpretazione: 11 personaggi su 15 non hanno un valore P9071. Questa query fornisce la prova quantitativa del nostro gap.

Query 4: archetipi già presenti in Wikidata

Questa query verifica quali termini narrativi esistono già come entità Wikidata e quali invece richiedono un namespace locale.

SELECT DISTINCT ?archetype ?archetypeLabel
WHERE {
  ?archetype rdfs:label ?archetypeLabel .
  FILTER(LANG(?archetypeLabel) = "en")
  FILTER(REGEX(?archetypeLabel, "tsundere|kuudere|dandere|yandere|antihero|mentor|rival|protagonist|villain|chosen one", "i"))
}
ORDER BY ?archetypeLabel
LIMIT 80

Query 5: estrazione delle proprietà di un singolo personaggio

Questa query è stata usata nella fase iniziale per osservare quali proprietà fossero già disponibili su un personaggio specifico. L'esempio usa Sasuke Uchiha (wd:Q1740).

SELECT ?property ?propertyLabel ?value ?valueLabel
WHERE {
  wd:Q1740 ?property ?value .
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en,it". }
}
ORDER BY ?propertyLabel
LIMIT 100

Uso metodologico: questa query mostra che Wikidata descrive bene aspetti identificativi ed enciclopedici, ma non sempre include informazioni narratologiche.

Query 6: confronto delle proprietà nel campione

Questa query estende l'estrazione delle proprietà a tutti i 15 personaggi del corpus, permettendo di confrontare la ricchezza informativa tra personaggi principali e secondari.

SELECT DISTINCT ?character ?characterLabel ?property ?propertyLabel ?value ?valueLabel
WHERE {
  VALUES ?character { wd:Q477948 wd:Q858432 wd:Q877964 wd:Q1061765 wd:Q1190653 wd:Q931 wd:Q1740 wd:Q203240 wd:Q1767 wd:Q1043344 wd:Q843545 wd:Q52863 wd:Q687038 wd:Q2328583 wd:Q138008 }
  ?character ?property ?value .
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en,it". }
}
ORDER BY ?characterLabel ?propertyLabel
LIMIT 500

Query 7: frequenza delle proprietà nel corpus

Questa è una delle query più importanti per dimostrare il gap: conta su quanti personaggi compare ogni proprietà.

SELECT ?property ?propertyLabel (COUNT(DISTINCT ?character) AS ?numberOfCharacters)
WHERE {
  VALUES ?character { wd:Q477948 wd:Q858432 wd:Q877964 wd:Q1061765 wd:Q1190653 wd:Q931 wd:Q1740 wd:Q203240 wd:Q1767 wd:Q1043344 wd:Q843545 wd:Q52863 wd:Q687038 wd:Q2328583 wd:Q138008 }
  ?character ?property ?value .
  SERVICE wikibase:label { bd:serviceParam wikibase:language "en,it". }
}
GROUP BY ?property ?propertyLabel
ORDER BY DESC(?numberOfCharacters)
LIMIT 50

Interpretazione: proprietà come label, descrizione, immagine, opera o genere tendono a essere più frequenti. P9071, invece, compare solo in pochi casi, confermando il gap sugli archetipi narrativi.

Query 8: confronto tra personaggi principali e secondari

Questa query divide manualmente il campione in personaggi principali e secondari, poi conta quanti hanno almeno un valore per P9071.

SELECT ?group (COUNT(DISTINCT ?character) AS ?characters) (COUNT(DISTINCT ?characterWithType) AS ?charactersWithP9071)
WHERE {
  VALUES (?character ?group) {
    (wd:Q477948 "main character") (wd:Q931 "main character") (wd:Q843545 "main character")
    (wd:Q858432 "secondary character") (wd:Q877964 "secondary character") (wd:Q1061765 "secondary character") (wd:Q1190653 "secondary character")
    (wd:Q1740 "secondary character") (wd:Q203240 "secondary character") (wd:Q1767 "secondary character") (wd:Q1043344 "secondary character")
    (wd:Q52863 "secondary character") (wd:Q687038 "secondary character") (wd:Q2328583 "secondary character") (wd:Q138008 "secondary character")
  }
  OPTIONAL { ?character wdt:P9071 ?characterType . BIND(?character AS ?characterWithType) }
}
GROUP BY ?group
